## 3. 分层职责

### 3.1 前端 `src/`

前端按业务功能组织，而不是把所有页面和组件堆在同一个目录。

每个 `features/<name>/` 可包含：

```text
accounts/
├── api.ts
├── components/
├── hooks/
├── model.ts
├── pages/
└── tests/
```

约束：

- `api.ts` 只能调用 `src/lib/tauri.ts`，不得直接散落 `invoke()`。
- 平台状态以 Rust 返回值为准，前端不模拟“在线”。
- 前端 Store 只保存 UI 状态和服务端快照，不保存密钥或 Session。
- 用户可见文案进入 i18n 文件。
- 消息正文避免进入浏览器控制台日志。

### 3.2 Tauri 命令 `commands/`

命令层负责：

- 参数反序列化和校验。
- 获取应用状态。
- 调用服务层。
- 把领域错误映射为稳定的前端错误结构。

命令层不应直接：

- 拼 SQL。
- 操作 Playwright DOM。
- 读取环境变量中的业务密钥。
- 实现复杂重试或状态机。

### 3.3 领域层 `domain/`

领域层保存与框架无关的核心类型：

- `Account`、`AccountStatus`
- `Conversation`
- `Message`、`MessageStatus`
- `OutboxItem`
- `AppError`、`ErrorCode`

该层应尽可能保持纯逻辑，便于单元测试。

### 3.4 服务层 `services/`

服务层编排存储、适配器和事件：

- `AccountService`：授权、恢复、状态变化和删除。
- `MessageService`：同步、去重、创建出站消息。
- `TranslationService`：语言检测、翻译、缓存和术语表。
- `OutboxWorker`：租约、发送、重试和结果落库。

### 3.5 适配器层 `adapters/`

- `traits.rs` 定义稳定的统一接口。
- `mock.rs` 用于开发和自动化测试。
- `official/` 放平台官方 API 连接器。
- `sidecar/` 只处理 Rust 与 Node Sidecar 的通信和生命周期。

平台特有字段应在适配器边界完成转换，避免渗透到 UI 和数据库公共字段。

### 3.6 存储层 `storage/`

- Repository 提供领域语义方法，不向上暴露 SQLx 查询细节。
- 多表状态变化使用事务。
- migration 文件不可在发布后修改，只能新增。
- 所有时间统一存 UTC Unix 毫秒，显示时再转本地时区。

### 3.7 安全层 `security/`

- `keychain.rs` 封装不同操作系统密钥链。
- `session_vault.rs` 负责密文格式、版本和轮换。
- `redaction.rs` 为日志和诊断包提供统一脱敏。
- 安全层 API 返回受控类型，避免敏感字节被意外格式化输出。

### 3.8 Sidecar `sidecar/`

建议使用 TypeScript 开发并构建为 JavaScript：

- `protocol/`：JSONL schema、解析、校验和错误响应。
- `runtime/`：浏览器池、上下文和资源回收。
- `platforms/`：平台页面检测与动作。
- `test/`：HTML fixture、协议测试和选择器测试。

Sidecar 的 `stdout` 只能输出协议消息，调试日志必须输出到 `stderr`。

## 4. 文件与命名约定

### 4.1 TypeScript

- 组件：`PascalCase.tsx`
- Hook：`useSomething.ts`
- 普通模块：`kebab-case.ts` 或项目统一采用 `camelCase.ts`
- 测试：`*.test.ts(x)`
- 禁止无说明的 `any`
- IPC 请求和响应必须有共享 schema 或生成类型

### 4.2 Rust

- 模块和文件：`snake_case`
- 类型：`PascalCase`
- 错误统一进入 `AppError`
- 公共函数返回 `Result<T, AppError>`
- 禁止在业务路径中使用 `unwrap()` 和 `expect()`
- 敏感结构不得派生或实现可泄露内容的 `Debug`

### 4.3 数据库

- 表名和字段名使用 `snake_case`
- 主键使用 UUID 或 ULID 字符串，并在全项目固定一种
- 外键明确指定删除策略
- 唯一约束用于实现幂等，不只依靠应用层查询

## 5. 配置与密钥

### 5.1 可提交的 `.env.example`

只放变量名和安全示例：

```dotenv
RUST_LOG=info
MULTICONNECT_DATA_DIR=
TRANSLATION_PROVIDER=mock
TRANSLATION_API_KEY=
```

### 5.2 规则

- `.env`、数据库、日志、Session 和浏览器 profile 必须进入 `.gitignore`。
- 生产 API Key 优先保存系统密钥链，不长期依赖 `.env`。
- 配置分为：编译配置、普通用户设置、敏感设置。
- 前端只获取允许展示的普通设置。

### 5.3 默认数据目录

通过 Tauri path API 获取，不手写用户目录：

- Windows：应用数据目录。
- macOS：Application Support 目录。
- Linux：XDG data/config 目录。

数据库、日志、临时文件、浏览器 profile 和导出文件应分目录管理。

## 6. IPC 开发约定

前端不直接传递任意 JSON。每个命令定义明确 DTO：

```ts
export interface SendMessageInput {
  accountId: string;
  conversationId: string;
  originalText: string;
  translatedText?: string;
  idempotencyKey: string;
}
```

统一错误：

```ts
export interface CommandError {
  code: string;
  message: string;
  retryable: boolean;
  requestId: string;
}
```

新增 IPC 的步骤：

1. 在 Rust 定义输入、输出和错误。
2. 增加服务层方法和测试。
3. 在 `commands/mod.rs` 注册。
4. 更新前端类型化封装。
5. 更新 `docs/api/tauri-commands.md`。
6. 增加成功、校验失败和服务失败测试。

## 7. 推荐脚本

最终命令以创建脚手架后的 `package.json` 和 `Cargo.toml` 为准。建议提供：

| 命令 | 作用 |
|:--|:--|
| `pnpm install` | 安装前端与工作区依赖 |
| `pnpm dev` | 启动前端开发服务器 |
| `pnpm tauri dev` | 启动桌面开发模式 |
| `pnpm test` | 前端与 Sidecar 测试 |
| `pnpm lint` | ESLint |
| `pnpm typecheck` | TypeScript 类型检查 |
| `cargo fmt --check` | Rust 格式检查 |
| `cargo clippy --all-targets -- -D warnings` | Rust 静态检查 |
| `cargo test` | Rust 测试 |
| `pnpm check` | 聚合全部本地检查 |

Windows 辅助脚本只能封装这些命令，不应成为唯一可运行入口。

## 8. CI 基线

每个提交至少执行：

1. Markdown 链接和格式检查。
2. TypeScript 格式、lint、类型和单元测试。
3. Rust fmt、clippy 和测试。
4. Sidecar 协议测试。
5. 依赖漏洞、许可证和密钥扫描。
6. 至少一个桌面平台的 debug 构建。

发布分支再执行多平台构建、签名、SBOM 和安装冒烟测试。

## 9. 测试目录约定

| 测试 | 位置 | 目标 |
|:--|:--|:--|
| React 单元/组件 | `src/**/tests/` | 组件行为和状态 |
| Rust 单元 | 对应模块内 `#[cfg(test)]` | 纯领域逻辑 |
| Rust 集成 | `src-tauri/tests/` | 数据库、命令、适配器 |
| Sidecar 单元 | `sidecar/test/` | 协议、页面 fixture |
| E2E | 后续 `e2e/` | 桌面关键路径 |

测试不得依赖个人账号、生产凭据或真实联系人。

## 10. 首批实现清单

### Step 1：工程脚手架

- 初始化 Git 仓库。
- 创建 Tauri 2 + React + TypeScript 工程。
- 建立 pnpm、Rust 工具链和 CI。
- 增加 mock adapter，使 UI 开发不依赖真实平台。

### Step 2：领域与存储

- 创建账号、会话、消息和 outbox 类型。
- 增加 SQLite migration 和 repository。
- 实现错误模型、结构化日志和脱敏。

### Step 3：安全与恢复

- 实现系统密钥链抽象。
- 实现 SessionVault 加解密和篡改测试。
- 实现应用启动恢复框架。

### Step 4：首个平台闭环

- 形成平台接入 ADR。
- 实现授权、恢复、同步和纯文本发送。
- 完成去重、重连、失败状态和端到端测试。

### Step 5：翻译与 UI

- 实现 mock 和首个真实翻译提供商。
- 完成翻译预览、原文保留和逐账号开关。
- 完成账号、会话、消息和设置页面。

## 11. 完成状态维护规则

功能状态统一在 issue、里程碑或专门的 `docs/status.md` 中维护，避免在多个文档重复手工打勾。

只有满足以下条件才能写“已实现”：

- 对应源码存在。
- 本地或 CI 可构建。
- 自动化测试通过。
- 验收标准有记录。
- 已知限制已写入文档。

设计文档中的接口名和目录仅为目标结构，实际实现发生变化时，应同步更新 ADR 和本文。
