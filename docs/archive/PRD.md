# MultiConnect 产品需求文档

> 产品名称：MultiConnect  
> 文档版本：v1.1  
> 文档状态：需求基线  
> 更新日期：2026-06-19  
> 当前实现状态：工程基线开发中；客户端 UI、Mock 翻译核心、自有控制后端、Rust WSS 客户端和通信契约已有代码，真实平台能力仍待实现与验收

## 1. 产品概述

### 1.1 产品定位

MultiConnect 是一套面向跨境客服、客户成功和已授权客户运营场景的多渠道消息工作台。产品计划在一个桌面端中统一管理 WhatsApp、Telegram 和 RCS 的账号状态、会话、翻译、联系人与任务，并在后续提供受控的 Web 远程管理能力。

### 1.2 核心价值

- 集中查看不同平台的账号在线状态和会话。
- 保存授权状态，减少重复登录。
- 对收发消息提供可审阅的双向翻译。
- 为客服团队提供统一、可追踪的处理流程。
- 对通知和批量任务实施同意校验、频率限制、退订和审计。

### 1.3 非目标

以下能力不属于产品目标：

- 绕过平台验证、封禁、反自动化或频率限制。
- 未经联系人同意发送营销或骚扰消息。
- 抓取、出售或跨租户共享联系人数据。
- 破解客户端、修改第三方 APK 或窃取登录凭据。
- 对平台明确禁止的自动化行为提供规避方案。

### 1.4 成功指标

MVP 上线后的建议观测指标如下：

| 指标 | MVP 目标 | 说明 |
|:--|:--|:--|
| 登录成功率 | ≥ 95% | 在平台服务正常且用户操作正确时 |
| Session 恢复成功率 | ≥ 90% | 不包含平台主动使会话失效 |
| 消息同步 P95 延迟 | ≤ 3 秒 | 从适配器捕获到本地 UI 展示 |
| 单条消息发送成功率 | ≥ 98% | 排除无效联系人和平台拒绝 |
| 崩溃后任务可恢复率 | 100% | 已入持久队列且未过期的任务 |
| 敏感信息明文落盘 | 0 | 包括令牌、Cookie、API Key |
| 自动化测试覆盖 | 核心领域逻辑 ≥ 80% | 不要求 UI 样式代码达到该比例 |

## 2. 目标用户与场景

| 用户 | 典型场景 | 核心需求 |
|:--|:--|:--|
| 跨境客服 | 多渠道处理售前、售后消息 | 会话聚合、翻译、快捷回复 |
| 客户成功团队 | 维护已签约客户 | 客户分组、跟进记录、审计 |
| 电商运营 | 向已订阅客户发送订单或活动通知 | 模板、定时任务、退订管理 |
| 小型团队管理员 | 管理多个渠道和坐席 | 权限、账号健康度、统计 |

## 3. 产品原则与约束

### 3.1 平台接入优先级

1. 优先使用官方 API，例如 WhatsApp Business Platform、Telegram Bot API 或经批准的客户端能力、RCS Business Messaging。
2. 若业务必须使用 Web 适配器，应为该适配器增加独立开关和风险提示，并完成平台条款评估。
3. Web 页面结构变化不得影响核心数据安全；适配器失败时应安全停机，不应无限重试。

### 3.2 数据原则

- 默认本地优先，Session 不上传远程服务。
- 翻译仅传输完成翻译所必需的文本，界面中明确提示第三方处理。
- 日志默认脱敏，不记录消息正文、Cookie、令牌和完整手机号。
- 删除账号时，用户可选择仅退出登录或同时删除关联本地数据。

### 3.3 状态定义

| 状态 | 含义 |
|:--|:--|
| 规划中 | 已定义需求，尚无实现 |
| 开发中 | 已有代码但未通过完整验收 |
| 已实现 | 代码、测试和验收记录齐备 |
| 暂缓 | 当前版本不实施 |

当前仓库已有可运行的工程骨架，但界面账号、消息和翻译结果仍包含演示数据。功能状态以源码、自动化测试和验收记录为准；真实平台登录、Session 恢复与消息收发仍为“规划中”。

## 4. MVP 范围

### 4.1 MVP 必须包含

- 桌面端基础框架。
- 一个平台通道的登录、状态恢复、会话列表和单条消息收发。
- 账号添加、重连、退出和删除。
- 本地加密存储与系统密钥链。
- 实时消息流、消息去重和失败提示。
- 中英双向翻译预览、确认发送和逐账号开关。
- 基础诊断日志和崩溃恢复。

### 4.2 MVP 不包含

- 多租户 Web 控制台。
- 大规模批量营销。
- AI 自动生成并直接发送消息。
- 无人工确认的自动回复。
- Telegram 与 RCS 同时接入；应在首个平台稳定后逐步增加。

## 5. 功能需求

### 5.1 账号管理

| ID | 需求 | 优先级 | 验收要点 |
|:--|:--|:--|:--|
| ACC-01 | 展示账号卡片 | P0 | 展示平台、名称、连接状态、最近同步时间和错误摘要 |
| ACC-02 | 添加账号 | P0 | 用户选择平台后进入明确的授权流程；取消不会留下残余数据 |
| ACC-03 | 恢复账号 | P0 | 应用重启后自动加载已保存授权并更新真实状态 |
| ACC-04 | 重连账号 | P0 | 支持手动重试；重试采用退避策略且可取消 |
| ACC-05 | 退出与删除 | P0 | 区分“退出平台”和“删除本地账号”；危险操作二次确认 |
| ACC-06 | 账号分组 | P2 | 可按团队、地区或业务线筛选 |

账号状态统一为：`initializing`、`awaiting_auth`、`online`、`degraded`、`offline`、`expired`、`error`。

### 5.2 平台授权与 Session

| ID | 需求 | 优先级 | 验收要点 |
|:--|:--|:--|:--|
| AUTH-01 | 用户可见的授权窗口 | P0 | 二维码、手机号或验证码流程清晰；不在后台偷偷登录 |
| AUTH-02 | 完整授权状态保存 | P0 | 保存平台恢复所需的 Cookie、Local Storage 等状态，而非只保存 Cookie |
| AUTH-03 | 加密持久化 | P0 | 使用认证加密；主密钥进入系统密钥链 |
| AUTH-04 | 过期检测 | P0 | 过期后停止发送，标记为 `expired` 并提示重新授权 |
| AUTH-05 | 授权撤销 | P1 | 用户可清除本地状态并尽可能撤销平台会话 |

### 5.3 会话与消息

| ID | 需求 | 优先级 | 验收要点 |
|:--|:--|:--|:--|
| MSG-01 | 会话列表 | P0 | 按最新消息排序，显示未读数和同步状态 |
| MSG-02 | 历史消息 | P0 | 分页加载；无重复、无明显乱序 |
| MSG-03 | 新消息监听 | P0 | 新消息在目标延迟内进入本地数据库和 UI |
| MSG-04 | 单条发送 | P0 | 发送状态包含排队、发送中、成功、失败和未知 |
| MSG-05 | 幂等与去重 | P0 | 平台消息 ID 或复合键保证重复事件不会重复展示 |
| MSG-06 | 失败重试 | P1 | 仅对可重试错误执行有限次数重试；用户可手动重试 |
| MSG-07 | 附件 | P2 | 后续支持图片和文件；MVP 先支持纯文本 |

### 5.4 翻译

| ID | 需求 | 优先级 | 验收要点 |
|:--|:--|:--|:--|
| TR-01 | 接收消息翻译 | P0 | 原文始终保留；翻译失败不影响原文展示 |
| TR-02 | 发送前翻译 | P0 | 展示译文预览，用户确认后才发送 |
| TR-03 | 语言检测 | P0 | 已是目标语言时不重复翻译 |
| TR-04 | 逐账号开关 | P0 | 配置持久化，切换后仅影响该账号 |
| TR-05 | 翻译提供商 | P1 | 通过后端适配器切换，不在前端保存密钥 |
| TR-06 | 缓存 | P1 | 按提供商、源语言、目标语言和文本哈希缓存 |
| TR-07 | 术语表 | P2 | 支持团队术语，修改后可版本化 |

翻译消息展示必须区分“平台原文”和“机器译文”，不得把译文伪装成对方原话。

### 5.5 联系人与合规任务

| ID | 需求 | 优先级 | 验收要点 |
|:--|:--|:--|:--|
| CON-01 | 联系人同步 | P1 | 每个平台账号独立存储映射，避免错误串号 |
| CON-02 | 导入导出 | P1 | CSV 导入提供预览、校验和错误报告 |
| CON-03 | 标签与分组 | P1 | 支持一个联系人多个标签 |
| CON-04 | 同意记录 | P1 | 保存同意来源、时间、范围和撤回状态 |
| JOB-01 | 通知任务 | P1 | 选择账号、已同意联系人、模板和发送时间 |
| JOB-02 | 频率限制 | P1 | 平台、账号、联系人三级限流 |
| JOB-03 | 退订 | P1 | 退订后立即阻止后续非必要消息 |
| JOB-04 | 紧急停止 | P1 | 用户可暂停单任务或全部任务 |
| JOB-05 | 审计日志 | P1 | 记录创建、批准、执行、暂停和失败原因 |

### 5.6 系统集成

| ID | 需求 | 优先级 | 验收要点 |
|:--|:--|:--|:--|
| SYS-01 | 系统托盘 | P1 | 最小化后可查看总体连接状态并安全退出 |
| SYS-02 | 通知 | P1 | 可关闭；通知内容遵循隐私设置 |
| SYS-03 | 开机启动 | P2 | 默认关闭，由用户主动启用 |
| SYS-04 | 更新 | P2 | 更新包签名校验；失败可回滚 |
| SYS-05 | 诊断包 | P1 | 导出脱敏日志、版本和状态，不包含凭据或正文 |

### 5.7 Web 远程控制台

该模块在桌面 MVP 稳定后实施。

| ID | 需求 | 优先级 |
|:--|:--|:--|
| WEB-01 | 查看账号和任务状态 | P2 |
| WEB-02 | 远程收发消息 | P2 |
| WEB-03 | 角色与账号范围权限 | P2 |
| WEB-04 | 审计与登录安全 | P2 |
| WEB-05 | 远程触发桌面授权流程 | P3 |

Web 控制台不得接收或存储平台 Session；远程命令由已配对桌面客户端执行。

## 6. 关键用户流程

### 6.1 添加账号

1. 用户点击“添加账号”并选择平台。
2. 应用展示平台接入方式和风险说明。
3. 用户在可见授权窗口完成授权。
4. 适配器验证登录状态并读取必要的账号摘要。
5. Session 加密落盘，账号进入 `online`。
6. 任一步骤失败时，应用清理临时上下文并显示可操作错误。

### 6.2 发送翻译消息

1. 用户输入原文。
2. 系统检测语言并请求翻译。
3. UI 同时展示原文和译文。
4. 用户可编辑译文并确认发送。
5. 消息进入持久发送队列。
6. 适配器回传平台结果，UI 更新最终状态。

### 6.3 Session 恢复

1. 应用启动并读取账号索引。
2. 从系统密钥链获取主密钥并解密 Session。
3. 适配器恢复上下文并验证平台登录状态。
4. 成功则开始同步；失败则标记 `expired` 或 `error`。
5. 恢复过程不得阻塞主窗口启动。

## 7. 非功能需求

### 7.1 性能基线

| 指标 | MVP 目标 |
|:--|:--|
| 冷启动到主界面 | ≤ 5 秒 |
| 消息 UI 更新 P95 | ≤ 3 秒 |
| 普通交互响应 P95 | ≤ 200 毫秒 |
| 本地数据库查询 P95 | ≤ 100 毫秒 |
| 空闲内存 | 建议 ≤ 500 MB（单账号、单平台） |

“8 GB 内存运行 50 个独立浏览器账号”暂不作为承诺。账号容量应通过实际压测确定，并在不同平台与接入方式下分别公布。

### 7.2 安全

- Session 使用 AES-256-GCM 或等价认证加密。
- 每份密文使用独立随机 nonce，禁止 nonce 重用。
- 主密钥存入 Windows Credential Manager、macOS Keychain 或 Linux Secret Service。
- 翻译 API Key 仅由可信后端或本地安全服务读取。
- IPC 命令进行参数校验和权限最小化。
- 远程通信使用 TLS，并采用设备配对、短期令牌和可撤销凭据。
- 依赖和安装包进入漏洞扫描、签名和软件物料清单流程。

### 7.3 可靠性

- 发送任务先持久化再执行。
- 进程崩溃后可从最后确认点恢复。
- 重连采用指数退避、随机抖动和最大尝试次数。
- 无法确认发送结果时标记为 `unknown`，不得盲目重发。
- 平台适配器互相隔离，单个平台故障不拖垮主进程。

### 7.4 可访问性与国际化

- 首期提供简体中文和英文。
- 键盘可完成核心操作。
- 关键状态不只依赖颜色表达。
- 用户界面文本使用 i18n 资源，不硬编码在组件中。

## 8. 数据与隐私

### 8.1 数据分类

| 级别 | 示例 | 处理要求 |
|:--|:--|:--|
| 高敏感 | Session、访问令牌、API Key | 加密、禁止日志、最小访问 |
| 敏感 | 消息正文、手机号、联系人名称 | 本地优先、可删除、日志脱敏 |
| 内部 | 任务配置、错误详情 | 权限控制、保留期限 |
| 普通 | 应用版本、匿名性能指标 | 用户可选择是否上报 |

### 8.2 数据生命周期

- 用户可按账号删除消息缓存和联系人。
- 默认日志保留 14 天，支持用户修改或立即清除。
- 删除账号后清理 Session、索引、临时文件和派生缓存。
- 远程备份必须由用户明确启用，并采用端到端或客户持有密钥的加密方案。

## 9. 版本规划

### Phase 0：工程基线（1–2 周）

- 创建项目脚手架、代码规范和 CI。
- 建立本地数据库、配置、日志和密钥管理。
- 确认首个平台的官方接入方式和合规要求。

### Phase 1：单平台 MVP（5–8 周）

- 完成一个平台的授权、恢复、会话与纯文本收发。
- 完成账号面板、消息流、翻译预览和错误处理。
- 完成单元、集成和最小端到端测试。

### Phase 2：可靠性与第二平台（4–6 周）

- 增加断线恢复、消息去重、诊断和性能优化。
- 在适配器契约稳定后接入第二个平台。
- 增加联系人和同意记录。

### Phase 3：受控任务（4–6 周）

- 模板、定时任务、限流、退订、审批和审计。
- 先以小规模通知场景验证，不承诺高吞吐营销群发。

### Phase 4：远程控制台（4–6 周）

- 设备配对、角色权限、实时状态和审计。
- 远程端不保存平台 Session。

## 10. 发布门槛

一个里程碑只有同时满足以下条件才能标记为“已完成”：

- 功能代码已合并并可重复构建。
- 自动化测试通过，关键路径有端到端验证。
- 安全检查无未处理的高危问题。
- 文档、配置示例和错误处理已更新。
- 有明确验收记录，且已知限制已列出。

MVP 发布前还必须完成：

- Session 加密和删除流程测试。
- 平台授权过期及恢复测试。
- 重复消息、断网、进程崩溃和磁盘空间不足测试。
- 翻译提供商不可用时的降级测试。
- 安装包签名和升级回滚测试。

## 11. 风险与缓解

| 风险 | 概率 | 影响 | 缓解 |
|:--|:--|:--|:--|
| 平台条款不允许 Web 自动化 | 高 | 高 | 优先官方 API；上线前法务与平台政策评估 |
| 页面结构变化导致适配器失效 | 高 | 高 | 适配器隔离、契约测试、快速停用开关 |
| Session 泄露 | 低 | 极高 | 系统密钥链、认证加密、日志扫描、最小权限 |
| 消息重复发送 | 中 | 高 | 幂等键、持久队列、未知状态不自动重试 |
| 多账号资源占用过高 | 中 | 中 | 压测、并发上限、空闲休眠、官方 API 优先 |
| 翻译错误造成业务误解 | 中 | 中 | 保留原文、发送前确认、术语表、免责声明 |
| 未经同意的消息触达 | 中 | 高 | 同意记录、退订、限流、审批和审计 |
| 第三方翻译服务数据跨境 | 中 | 高 | 提供商披露、区域配置、本地模型选项 |

## 12. 待决策事项

1. 首个平台采用官方 API 还是受限 Web 适配器？
2. 产品首期面向个人单机还是团队多坐席？
3. 本地消息默认保留多久？
4. 翻译提供商和数据处理区域如何选择？
5. 是否允许远程控制台读取消息正文？
6. 首个受控任务场景是订单通知、客服回访还是营销订阅？
7. Tauri Sidecar 中 Node.js 与浏览器运行时如何打包和更新？

这些决策应在 Phase 0 结束前形成 ADR（架构决策记录）。
