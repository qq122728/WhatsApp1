# MultiConnect 技术设计文档

> 文档版本：v1.1  
> 文档状态：设计基线  
> 更新日期：2026-06-19  
> 实现状态：工程基线开发中；设备注册与 WSS v1 控制通道已有可验证闭环，平台适配与持久安全存储仍以目标设计为准

当前 WhatsApp 适配已完成第一阶段：Node.js Sidecar 使用 Playwright `launchPersistentContext` 启动可见 Chrome，并为每个账号分配独立 `userDataDir`。当前仅负责登录与恢复状态，不包含消息抓取或自动发送。

## 1. 设计目标

本设计面向桌面优先的多渠道消息工作台，重点解决：

- 平台适配器与业务层解耦。
- 授权状态安全保存和可恢复。
- 消息收发的幂等、可靠性与可审计性。
- 翻译提供商可替换，API Key 不暴露给前端。
- 单个平台故障不影响整个客户端。
- 后续可扩展 Web 远程控制台，但平台 Session 始终留在桌面端。

## 2. 技术选型

| 模块 | 建议方案 | 说明 |
|:--|:--|:--|
| 桌面框架 | Tauri 2.x | 小体积、Rust 后端、原生系统能力 |
| 前端 | React + TypeScript + Vite | 组件生态成熟，适合状态驱动 UI |
| UI 样式 | Tailwind CSS 或组件库 | 项目启动时二选一并固定 |
| 本地后端 | Rust + Tokio | 负责领域服务、存储、密钥、队列和进程管理 |
| 本地数据库 | SQLite + SQLx | WAL 模式、迁移可追踪 |
| Web 自动化适配器 | Node.js Sidecar + Playwright | 仅用于经评估允许的 Web 接入 |
| 官方 API 适配器 | Rust HTTP/WebSocket 客户端 | 优先于 Web 自动化 |
| 加密 | AES-256-GCM | 认证加密，密钥存系统密钥链 |
| 日志 | tracing + rolling file | 结构化、分级、脱敏 |
| 远程后端 | FastAPI 或 Rust 服务 | Phase 4 再决策 |

### 2.1 关键修正

- “无痕模式”不适合作为 Session 持久化方案。Web 适配器应使用每账号独立的持久化用户目录或完整 `storageState`。
- 只保存 Cookie 通常不足以恢复现代 Web 应用；还需评估 Local Storage、IndexedDB、Service Worker 和平台令牌。
- 翻译 API 不应由 React 前端直接调用，否则 API Key 容易泄露。调用应经过 Rust 服务或受控远程后端。
- 一个账号一个 Browser Context 不等于可以低成本运行大量账号。容量必须通过真实压测确认。

## 4. 进程与模块边界

### 4.1 React UI

只负责展示和用户交互：

- 不直接访问平台网络接口。
- 不读取 Session、主密钥或翻译 API Key。
- 不自行判断账号真实在线状态。
- 通过类型化 Tauri 命令读取快照，通过事件订阅增量变化。

### 4.2 Rust Application Core

Rust 进程是本地可信边界，负责：

- 领域模型和状态机。
- SQLite 读写与事务。
- Session 加密和系统密钥链访问。
- 平台适配器生命周期。
- 消息队列、幂等、重试和审计。
- 翻译适配器、缓存和密钥读取。
- 日志脱敏、诊断包和系统集成。

### 4.3 Platform Adapter

所有平台统一实现下列逻辑接口：

```rust
#[async_trait]
pub trait PlatformAdapter {
    async fn begin_auth(&self, account_id: AccountId) -> Result<AuthChallenge>;
    async fn complete_auth(&self, account_id: AccountId) -> Result<AccountProfile>;
    async fn restore(&self, account_id: AccountId, session: SecretBytes) -> Result<()>;
    async fn health(&self, account_id: AccountId) -> Result<AdapterHealth>;
    async fn list_conversations(&self, account_id: AccountId, cursor: Option<String>)
        -> Result<Page<Conversation>>;
    async fn sync_messages(&self, account_id: AccountId, cursor: Option<String>)
        -> Result<MessageBatch>;
    async fn send_message(&self, request: SendRequest) -> Result<SendReceipt>;
    async fn logout(&self, account_id: AccountId) -> Result<()>;
}
```

具体签名可在实现时调整，但领域层不得依赖 Playwright DOM 选择器。

### 4.4 Playwright Sidecar

Sidecar 只用于 Web 适配器：

- 每个账号使用独立上下文或持久化 profile。
- 标准输出只写协议 JSON，不混入普通日志。
- 普通日志写标准错误，由 Rust 收集并脱敏。
- Sidecar 崩溃由 Supervisor 发现并有限重启。
- 平台选择器、页面检测和脚本集中在平台目录中。
- 浏览器可见授权与后台同步应使用不同运行模式。

## 5. 核心状态机

### 5.1 账号状态

```text
initializing
    ├─→ awaiting_auth ─→ online
    ├─→ online
    └─→ error

online ─→ degraded ─→ online
   │          ├──────→ offline
   │          └──────→ expired
   ├───────────────→ offline
   └───────────────→ expired

offline ─→ initializing
expired ─→ awaiting_auth
error   ─→ initializing
```

状态变化必须包含 `reason_code`、`occurred_at` 和可选的脱敏错误摘要。

### 5.2 出站消息状态

```text
draft → queued → sending → sent
                    ├────→ failed
                    └────→ unknown
```

- `failed`：平台明确拒绝或达到重试上限。
- `unknown`：请求可能已提交，但未得到可靠确认；默认禁止自动重发。
- 每条出站消息使用稳定的本地幂等键。

## 6. 数据模型

建议 SQLite 开启 `WAL`、`foreign_keys=ON` 和合理的 `busy_timeout`。

### 6.1 核心表

|:--|:--|:--|

| connector_type | TEXT | official_api / web_adapter |
| display_name | TEXT | 账号显示名 |
| external_id | TEXT NULL | 平台账号标识，必要时加密或哈希 |
| status | TEXT | 统一账号状态 |
| status_reason | TEXT NULL | 错误码 |
| translation_enabled | INTEGER | 0/1 |
| created_at | INTEGER | Unix 毫秒 |
| updated_at | INTEGER | Unix 毫秒 |
| last_seen_at | INTEGER NULL | 最近健康时间 |

#### session_records

| 字段 | 类型 | 说明 |
|:--|:--|:--|
| account_id | TEXT PK/FK | 所属账号 |
| format_version | INTEGER | 密文格式版本 |
| cipher_text | BLOB | 加密后的 Session |
| nonce | BLOB | 每次写入独立随机值 |
| aad | BLOB | 账号 ID、平台和版本等附加认证数据 |

#### conversations

| 字段 | 类型 | 说明 |
|:--|:--|:--|
| id | TEXT PK | 本地 ID |
| account_id | TEXT FK | 所属账号 |
| platform_conversation_id | TEXT | 平台会话 ID |

#### messages

| 字段 | 类型 | 说明 |
|:--|:--|:--|
| id | TEXT PK | 本地 ID |
| account_id | TEXT FK | 所属账号 |
| conversation_id | TEXT FK | 所属会话 |
| platform_message_id | TEXT NULL | 平台消息 ID |
| idempotency_key | TEXT | 本地幂等键 |

唯一索引优先使用 `(account_id, platform_message_id)`；平台无稳定 ID 时使用规范化复合指纹。

#### outbox

| 字段 | 类型 | 说明 |
|:--|:--|:--|
| id | TEXT PK | 队列项 ID |
| message_id | TEXT FK | 出站消息 |
| attempts | INTEGER | 已尝试次数 |
| next_attempt_at | INTEGER | 下一次执行时间 |
| lease_until | INTEGER NULL | Worker 租约 |
| last_error_code | TEXT NULL | 最近错误 |

#### translation_cache

缓存键至少包含：提供商、模型/版本、源语言、目标语言、术语表版本和原文 SHA-256。缓存中不得包含 API Key。

#### audit_logs

记录用户或系统执行的敏感操作，不保存 Session 和完整消息正文。

### 6.2 数据迁移

- 每次 schema 变化新增不可变迁移文件。
- 应用启动先备份元数据，再执行迁移。
- 迁移失败时阻止写入并给出恢复入口。
- 不允许生产版本自动降级 schema。

## 7. Session 安全设计

### 7.1 密钥层级

1. 首次启动生成 256 位随机主密钥。
2. 主密钥存入操作系统密钥链，数据库只保存密钥引用版本。
3. 每个 Session 使用随机 96 位 nonce 进行 AES-256-GCM 加密。
4. AAD 包含账号 ID、平台和密文格式版本，防止密文被交换到其他账号。

### 7.2 原子写入

若 Session 保存为独立文件：

1. 写入同目录临时文件。
2. 刷新文件内容。
3. 原子替换目标文件。
4. 设置仅当前用户可读权限。

若保存在 SQLite 中，应在单事务中更新密文、nonce 和版本。

### 7.3 密钥不可用

- 密钥链被锁定时，不尝试用弱密钥降级。
- UI 提示用户解锁系统密钥链。
- 密钥永久丢失时，只能删除不可解密 Session 并重新授权。

## 8. Rust 与 Sidecar 协议

### 8.1 传输

- stdin/stdout 使用 UTF-8 JSON Lines，一行一个完整对象。
- 每个请求必须包含唯一 `requestId`。
- Rust 为请求设置超时并维护 pending map。
- 消息大小设置上限，建议初始为 1 MiB；二维码等大数据优先使用临时文件或受控二进制通道。
- 协议初始化时协商 `protocolVersion`。

### 8.2 请求

```json
{
  "protocolVersion": 1,
  "requestId": "01J...",
  "method": "adapter.beginAuth",
  "params": {
    "accountId": "01J...",
    "platform": "whatsapp"
  }
}
```

### 8.3 成功响应

```json
{
  "protocolVersion": 1,
  "requestId": "01J...",
  "ok": true,
  "result": {
    "status": "awaiting_auth"
  }
}
```

### 8.4 错误响应

```json
{
  "protocolVersion": 1,
  "requestId": "01J...",
  "ok": false,
  "error": {
    "code": "AUTH_TIMEOUT",
    "message": "Authorization timed out",
    "retryable": true
  }
}
```

### 8.5 事件

```json
{
  "protocolVersion": 1,
  "event": "message.received",
  "sequence": 1842,
  "data": {
    "accountId": "01J...",
    "platformMessageId": "abc123"
  }
}
```

Rust 收到事件后先落库，再通知 UI。UI 事件不是事实来源，重载页面时必须从数据库恢复快照。

## 10. 消息同步与去重

### 10.1 入站流程

1. 适配器产生原始平台事件。
2. 映射为统一消息模型并校验字段。
3. 在事务中执行去重、写消息、更新会话游标和未读数。
4. 提交成功后发布 UI 事件。
5. 如启用翻译，异步创建翻译任务；翻译失败不回滚原消息。

### 10.2 去重策略

优先级从高到低：

1. 平台稳定消息 ID。
2. 平台事件 ID。
3. 账号、会话、方向、发送方、时间窗口和内容哈希组成的复合指纹。

复合指纹可能误判，必须记录策略版本，方便迁移和诊断。

### 10.3 出站流程

1. `message_send` 在数据库事务中创建消息和 outbox 项。
2. Worker 获取带租约的队列项。
3. 执行账号状态、同意记录和频率限制校验。
4. 调用适配器并保存回执。
5. 根据结果标记 `sent`、`failed` 或 `unknown`。
6. 仅对明确可重试错误使用指数退避。

## 11. 健康检查与重连

### 11.1 健康分层

- 进程健康：Sidecar 是否存活、协议是否可用。
- 账号健康：授权是否有效、平台连接是否正常。
- 同步健康：游标是否持续推进、事件是否积压。
- 队列健康：待发送数量、最老任务年龄、失败率。

### 11.2 重连策略

- 建议退避：1、2、5、10、30、60 秒，最大 5 分钟。
- 增加 0–20% 随机抖动，避免多个账号同时重连。
- 连续失败达到阈值后进入 `offline`，等待用户或低频探测。
- 认证过期直接进入 `expired`，不继续自动重试。
- 应用退出时取消重连任务并等待关键写入完成。

## 12. 翻译服务

### 12.1 接口

```rust
#[async_trait]
pub trait TranslationProvider {
    async fn detect_language(&self, text: &str) -> Result<DetectedLanguage>;
    async fn translate(&self, request: TranslationRequest) -> Result<TranslationResult>;
}
```

### 12.2 流程

- 接收端：消息落库 → 创建低优先级翻译任务 → 写回译文 → UI 增量更新。
- 发送端：用户请求预览 → 获取译文 → 用户确认或编辑 → 原文和最终译文一起保存。
- 失败策略：超时或配额不足时保留原文，并允许用户重试或原文发送。

### 12.3 隐私

- 调用第三方前显示提供商及数据处理提示。
- 可配置不翻译指定会话。
- 日志只记录请求 ID、字符数、耗时和错误码，不记录正文。

## 13. 错误模型

统一错误码建议：

| 类别 | 示例 | 默认可重试 |
|:--|:--|:--|
| AUTH | `AUTH_TIMEOUT`、`AUTH_EXPIRED` | 部分 |
| NETWORK | `NETWORK_TIMEOUT`、`DNS_FAILURE` | 是 |
| PLATFORM | `PLATFORM_RATE_LIMITED`、`PLATFORM_REJECTED` | 部分 |
| ADAPTER | `ADAPTER_CRASHED`、`SELECTOR_MISMATCH` | 部分 |
| STORAGE | `DB_LOCKED`、`DISK_FULL` | 部分 |
| CRYPTO | `KEYCHAIN_LOCKED`、`DECRYPT_FAILED` | 否 |
| TRANSLATION | `TRANSLATION_QUOTA`、`TRANSLATION_TIMEOUT` | 部分 |
| VALIDATION | `INVALID_ARGUMENT` | 否 |

面向用户的消息与内部错误详情分离。内部堆栈不得直接展示或上传。

## 14. 日志与可观测性

- 使用结构化日志字段：`request_id`、`account_id_hash`、`adapter`、`error_code`。
- 禁止记录 Cookie、令牌、密钥、二维码内容和完整消息正文。
- 手机号、用户名和账号 ID 默认哈希或部分掩码。
- 本地日志按大小和日期滚动，默认保留 14 天。
- 诊断包生成前执行第二次敏感信息扫描。
- 关键指标：同步延迟、重连次数、发送成功率、队列年龄、翻译耗时。

## 15. 安全边界

### 15.1 Tauri

- capabilities 只开放所需命令和窗口。
- 禁止 UI 任意执行 Shell、读取任意路径或打开任意 URL。
- CSP 限制脚本与网络来源。
- 外部链接交给系统浏览器，并校验协议。

### 15.2 Sidecar

- 固定可执行文件和资源路径。
- 发布包校验 Sidecar 完整性。
- 不把用户输入拼接为脚本或选择器执行。
- 限制临时文件目录和权限。

### 15.3 供应链

- 锁定 Rust、Node 和浏览器依赖版本。
- CI 执行依赖审计、许可证检查和密钥扫描。
- 发布产物签名，生成 SBOM。

## 16. 测试策略

### 16.1 单元测试

- 账号和消息状态机。
- 加密往返、错误密钥和密文篡改。
- 去重指纹和幂等逻辑。
- 重试分类、退避和限流。
- 翻译缓存键和术语表版本。

### 16.2 集成测试

- SQLite 迁移和事务。
- 系统密钥链抽象，CI 使用测试实现。
- Rust 与 Sidecar 协议、超时、乱序响应和崩溃。
- 模拟平台适配器的登录、同步和发送。

### 16.3 端到端测试

- 首次启动与添加账号。
- 授权取消、超时、成功和过期。
- 收消息、翻译、发送、重启恢复。
- 断网、Sidecar 崩溃、磁盘满和密钥链锁定。

真实平台测试应使用专用测试账号，避免 CI 依赖生产账号。

## 17. 构建与发布

- 开发环境使用本机 Node.js 和 Playwright 浏览器。
- 发布包需明确是否内置 Node.js 与浏览器运行时；不能假设用户已安装。
- Sidecar 的平台、架构和版本必须与 Tauri 构建矩阵匹配。
- Windows、macOS 和 Linux 分别完成签名、公证或包验证。
- 更新使用签名清单，并保留失败回滚能力。

## 18. 实施顺序

1. 建立工作区、CI、格式化、lint 和测试命令。
2. 实现领域模型、SQLite 迁移和测试适配器。
3. 实现 SessionVault 和系统密钥链。
4. 完成账号状态机和适配器契约。
5. 选择一个平台完成授权、恢复、同步和发送闭环。
6. 完成 outbox、去重、重连和诊断。
7. 集成翻译服务。
8. UI 完成账号、会话、消息和设置页面。
9. 通过发布门槛后再接入第二个平台。
